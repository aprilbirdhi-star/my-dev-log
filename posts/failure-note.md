# 📚 실패노트: 개발 중 마주한 9가지 핵심 기술 의문점 (학습 로그)

견고한 서비스를 개발하고 예기치 못한 시스템 오류를 최소화하려면 웹의 동작 원리와 컴퓨터 사이언스(CS) 기초를 온전히 내 것으로 만들어야 합니다. 

이 일지는 제가 개발을 해오며 머릿속으로 집요하게 파고들었던 **9가지 핵심 기술 의문점**들을 정리하고 복기한 공부 기록입니다.

---

## 🔒 Web Security & Authentication

### 1. 세션, 쿠키, 토큰의 차이가 뭘까?
* **쿠키(Cookie)**: 브라우저에 저장되는 단순한 텍스트 데이터 파일입니다. 매 요청마다 헤더에 실려 서버로 전송되나, 위변조가 쉬워 민감한 정보 보관에는 위험합니다.
* **세션(Session)**: 쿠키의 보안 문제를 해결하기 위해 도입되었습니다. 유저 식별 정보는 서버 메모리나 DB에 저장하고, 브라우저에는 임의의 Session ID(쿠키)만 넘겨 인증합니다. 서버의 메모리 부담이 커질 수 있습니다.
* **토큰(Token - JWT)**: 세션의 서버 확장성 문제를 보완합니다. 서버는 유저 정보를 암호화 서명한 토큰(JWT)을 발급하고, 서버 세션 저장소 없이 브라우저가 토큰을 들고 다닙니다. 무상태(Stateless) 아키텍처에 매우 유리합니다.

### 2. SQL Injection, XSS, CSRF는 뭘까?
* **SQL Injection**: 유저 입력창에 악의적인 SQL 구문을 주입해 DB 정보를 유출하거나 조작하는 공격입니다. (방어: ORM 사용 및 Parameterized Query 필수)
* **XSS (Cross-Site Scripting)**: 웹페이지에 악성 자바스크립트를 삽입하여 다른 유저의 브라우저에서 실행되게 만드는 공격입니다. 쿠키 탈취 등에 악용됩니다. (방어: HTML Escape 처리 및 쿠키 HttpOnly 옵션 필수)
* **CSRF (Cross-Site Request Forgery)**: 유저가 로그인된 상태에서 해커가 조작한 링크를 클릭하게 만들어, 유저의 권한으로 서버에 악의적인 요청을 보내게 만드는 공격입니다. (방어: CSRF Token 인증 도입 필수)

### 3. 비밀번호 해싱(Hashing)은 무슨 뜻이야?
* 비밀번호를 평문 그대로 DB에 저장하면 해킹 시 치명적입니다. 이를 복호화가 불가능한 일방향 함수로 암호화하여 저장하는 행위가 **해싱**입니다.
* 동일한 비밀번호는 항상 동일한 해시값을 갖기 때문에 레인보우 테이블(해시 사전) 공격에 취약합니다. 이를 막기 위해 유저마다 무작위 문자열을 소금 치듯 덧붙여 해싱하는 **솔팅(Salting)** 및 여러 번 반복 해싱하는 **키 스트레칭(Key Stretching)**을 적용해 방어합니다. (예: `bcrypt`, `scrypt` 알고리즘)

## ⚙️ Operating System & Performance

### 4. 프로세스와 스레드의 차이는 뭘까?
* **프로세스(Process)**: 운영체제로부터 독립된 메모리 영역(Code, Data, Stack, Heap)을 할당받아 실행 중인 프로그램의 인스턴스입니다. 한 프로세스가 죽어도 다른 프로세스에 영향을 주지 않습니다.
* **스레드(Thread)**: 프로세스 내부에서 실행 흐름의 단위입니다. 프로세스가 가진 메모리 영역을 공유(Stack만 독립적)하기 때문에 통신 속도가 빠르지만, 하나의 스레드 장애가 전체 프로세스를 죽일 수 있습니다.

### 5. 락을 잘못 잡으면 왜 데드락이 발생하는 걸까?
* 멀티스레드 환경에서 공유 자원의 일관성을 유지하기 위해 자원을 잠그는 행위가 **락(Lock)**입니다.
* **데드락(Deadlock, 교착상태)**은 두 개 이상의 프로세스/스레드가 서로가 가진 자원의 락이 풀리기만을 무한히 대기하는 상태를 말합니다. 예를 들어 스레드 A가 자원 1을 잡고 자원 2를 원하며, 스레드 B가 자원 2를 잡고 자원 1을 원할 때 발생합니다. 자원 획득 순서를 일치시켜 예방해야 합니다.

## 🛠️ Software Engineering Practise

### 6. 배포 후 문제 생기면 원인 추적은 어떻게 하는걸까?
* 실서버 배포 후 에러가 났을 때 원인 분석(RCA)을 하려면 시스템 로깅과 추적 인프라가 갖춰져 있어야 합니다.
* **추적 기법**:
  1. **어플리케이션 에러 트래킹**: Sentry 등 APM 도구를 연동해 에러 스택 트레이스와 발생 환경 자동 수집.
  2. **중앙 집중형 로깅**: 로그 파이프라인(ELK Stack 등)을 통해 분산 서버들의 로그를 통합 조회.
  3. **모니터링**: CPU/Memory 경고 슬랙 알림 설정.

### 7. 모듈 분리는 뭘까?
* 거대하고 복잡한 소스 코드를 기능과 역할에 따라 독립적인 작은 단위(모듈)로 쪼개어 관리하는 아키텍처 설계입니다.
* **핵심 기준**: **응집도는 높이고, 결합도는 낮춘다 (High Cohesion, Low Coupling)**. 특정 기능을 변경했을 때 다른 모듈이 영향을 받지 않도록 설계 경계를 나누는 것이 기본입니다.

### 8. 에러 핸들링은 어떻게 하는거지?
* 에러가 났을 때 프로그램이 비정상 종료되는 것을 막고, 유저에게 적절한 안내를 보여주는 예외 처리 기법입니다.
* **원칙**:
  1. **예외 던지기와 잡기**: 하위 레이어에서 발생한 에러를 무작정 무시하지 않고 상위 레이어로 전파(`throw`)한 후 공통 컨트롤러/미들웨어에서 통합 처리(`try-catch`).
  2. **명확한 피드백**: 유저에게는 추상적이고 안전한 에러 메시지(500 서버 오류 등)를 보여주고, 내부 로그에는 에러 원인이 담긴 상세 스택 트레이스를 기록합니다.

## 🗺️ Roadmap & Study Guide

### 9. 자료구조, OS, 네트워크, DB, 컴파일러 공부는 어떻게 하지?
* CS 기초는 책으로만 암기하면 금방 휘발됩니다. 실습 중심의 학습 로드맵을 권장합니다.
  * **자료구조/알고리즘**: 백준/리트코드를 풀며 손 코딩으로 구조화(배열, 링크드리스트, 트리 등)를 직접 구현해보기.
  * **네트워크**: `Wireshark` 툴로 실제 패킷을 뜯어보거나, Python/Java로 소켓 통신 채팅 프로그램 만들기.
  * **데이터베이스**: 로컬에 MySQL 등을 세우고 더미 데이터 100만 개를 생성해 인덱스 여부에 따른 쿼리 성능 개선 측정해보기.
  * **OS/컴파일러**: 간단한 어셈블리어 프로그래밍이나 미니 C 컴파일러 빌드 튜토리얼 구현해보기.
