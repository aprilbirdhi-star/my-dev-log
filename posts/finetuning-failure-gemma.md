# 🧠 실패노트: Gemma-2-9B와 Unsloth의 float32 강제 전환 충돌 (Connect AI OS 파인튜닝 실패)

안녕하세요! 이루다입니다.

어제 Qwen3-0.6B 모델 파인튜닝 실패에 이어, 오늘(7/8)은 조금 더 똑똑한 두뇌를 합치기 위해 **`google/gemma-2-9b-it`** 모델로 재도전했습니다. Connect AI OS 환경을 활용해 SFT(Supervised Fine-Tuning) 데이터를 빌드하고 학습을 시도했지만, 또 한 번 장벽에 부딪혔습니다.

실패가 누적될수록 제 머릿속 지식도 점점 파인튜닝되는 느낌입니다. 인공지능 개발자에게 '실패'라는 단어는 어울리지 않는 것 같습니다. 어쩌면 **'내 머릿속 파인튜닝 중'**이라는 표현이 더 적확할지도 모르겠네요. 

어제는 7시간 동안 코랩의 재생 버튼만 무지성으로 누르며 씨름했다면, 오늘은 제이슨(JSON) 데이터의 구조와 내용을 하나하나 뜯어보며 집중하는 저를 발견했습니다. AI 코딩 에이전트인 안티그래비티(Antigravity)와 친해지는 데 4개월 정도 걸린 것 같은데, 어제의 치열한 삽질 덕분인지 오늘부터는 코랩과 확실히 친해진 기분입니다.

하지만 오픈소스 AI 세상에 진짜 공짜는 없더군요. 남의 지식을 내 것으로 만들기 위해서는 실패하고, 다시 고쳐가는 튜닝 과정이 필수적입니다. 오늘 마주한 두뇌 합치기 실패 원인과 이를 극복할 방법을 공유합니다.

## 📸 마주한 에러 로그 (Connect AI OS 분석 결과)

Connect AI OS의 디버깅 셀(`SEb-EqTjDYdK`)에서 발생한 `ValueError` 메시지는 다음과 같았습니다:

> **Unsloth: Using float16 precision for gemma2 won't work! Using float32.**
> **ValueError: Some modules are dispatched on the CPU or the disk. Make sure you have enough GPU RAM to fit the quantized model.**

## ❓ 왜 이런 문제가 발생했을까요?

### 1. Gemma 2 모델의 아키텍처 특성 (Logit Soft-Capping)
Google의 Gemma 2 모델은 성능을 극대화하기 위해 '어텐션 로짓 소프트 캐핑(Attention Logit Soft-Capping)'이라는 기술을 사용합니다. 이 기술의 부작용으로, 학습을 반정밀도인 `float16`으로 진행하면 숫자가 너무 커지거나 작아져 연산 값이 유실되는 **수치적 불안정성(NaNs 발생)**이 나타납니다.
이 때문에 Unsloth 엔진은 Gemma 2 모델을 로드할 때 `float16`을 강제로 막고, 안정적인 `bfloat16` 또는 단정밀도인 `float32`로 로딩하게 만듭니다.

### 2. Tesla T4 GPU의 bfloat16 미지원
코랩 무료 버전에서 제공하는 **Tesla T4 GPU**는 출시된 지 다소 시간이 지난 아키텍처라 현대적인 `bfloat16`(Brain Float 16) 데이터 포맷을 하드웨어 레벨에서 지원하지 않습니다. 
결국 Unsloth는 `bfloat16` 대신 연산량이 2배인 `float32` 정밀도를 강제하게 되었습니다.

### 3. VRAM 용량 초과와 오프로드 에러
90억 개의 매개변수(9B)를 가진 모델을 `float32`로 로드하면, 단순 모델 로딩에만 약 **36GB의 VRAM**이 필요합니다. 아무리 4비트 양자화를 적용했더라도 학습을 위해 메모리에 가중치들이 풀리는 과정에서 T4 GPU의 한계인 **15GB VRAM**을 훌쩍 넘어가 버립니다.
이때 PyTorch는 남는 가중치들을 시스템 RAM(CPU)이나 로컬 디스크로 밀어내려(`Offload`) 하지만, Unsloth 학습 엔진은 오프로드된 상태에서의 파인튜닝을 지원하지 않아 에러를 내며 멈추게 된 것입니다.

## 💡 이 문제는 어떻게 해결해야 할까요? (액션 플랜)

현재 Connect AI OS와 무료 코랩(T4 GPU)을 사용 중이라면 선택할 수 있는 해결 방법은 크게 세 가지입니다.

### 방법 A. 베이스 모델 변경 (가장 추천 ⭐️)
현재 무료 GPU 환경에서 두뇌 합치기 학습을 완료하고 로컬 엔진에 올리려면, `float16` 학습을 안정적으로 지원하면서 크기가 적당한 다른 베이스 모델로 바꾸는 것이 가장 빠르고 확실한 해결책입니다.
* **추천 모델 1: `Qwen/Qwen2.5-7B-Instruct` 또는 `Qwen/Qwen2.5-1.5B-Instruct`**
  * Qwen2.5 모델은 Unsloth와 T4 GPU에서 `float16`으로 완벽하게 파인튜닝이 가능하며 한국어 성능도 매우 훌륭합니다.
* **추천 모델 2: `meta-llama/Meta-Llama-3-8B-Instruct`**
  * 8B 크기로 Gemma 2 9B와 성능이 대등하며, T4 GPU에서 LoRA 학습이 매우 원활히 진행됩니다.
* **추천 모델 3: `google/gemma-2-2b-it`**
  * Gemma 2의 성능을 꼭 경험하고 싶다면 크기가 훨씬 작은 **2B(20억 매개변수)** 모델을 선택하는 것도 방법입니다. `float32`로 변환되더라도 2B 크기이기 때문에 T4 VRAM 내에 안착할 가능성이 높습니다.

### 방법 B. 코랩 프로(Colab Pro) GPU 활용
만약 Gemma 2 9B 모델을 꼭 학습시키고 싶다면, 코랩 프로 결제를 통해 **L4 GPU**나 **A100 GPU** 런타임을 사용해야 합니다.
* L4 및 A100 GPU는 `bfloat16` 연산을 네이티브로 지원하므로, Unsloth가 `float32` 대신 `bfloat16`으로 가볍게 로딩하여 T4와 달리 메모리 터짐 없이 안정적으로 학습을 끝마칠 수 있습니다.

### 방법 C. 데이터셋 구조와 내 파일 구성 이해하기
이루다님께서 "내 파일이지만 어떤 식으로 구성되었는지 잘 몰라서 진도를 못 나가겠다"고 하신 부분은 지극히 당연한 고민입니다. 
오픈소스 데이터를 다룰 때는 먼저 데이터를 눈으로 확인해야 합니다.
1. Connect AI OS에서 생성된 **`aprilbirdhi-star` 데이터셋 JSONL 파일**을 로컬 텍스트 에디터(VS Code 등)로 열어봅니다.
2. 각 줄이 `{"instruction": "...", "input": "...", "output": "..."}` 혹은 `{"messages": [{"role": "user", "content": "..."}, {"role": "assistant", "content": "..."}]}` 구조 중 어떤 대화 포맷으로 작성되어 있는지 직접 눈으로 파악합니다.
3. 사용 중인 코랩 노트북의 데이터 로더(`load_dataset` 및 `FormattingPrompts`)가 해당 JSONL의 키(Key) 이름들과 일치하는지 대조합니다. 이 구조 매핑만 올바르게 일치시켜도 데이터 구조 분석은 끝납니다.
