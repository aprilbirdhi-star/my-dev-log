## 🔍 "메모리 용량 부족"의 진짜 의미는 무엇일까?

학습 도중 마주한 메모리 부족 에러는 **로컬 C드라이브 용량 부족이 아닙니다.** Google Colab은 구글의 클라우드 가상 머신(VM)에서 돌아가기 때문에, 제 로컬 컴퓨터의 저장공간(44.1 GB)과는 무관합니다. 

원인은 크게 **세 가지 시스템 메모리 부족** 중 하나일 가능성이 매우 높습니다.

```
💡 LLM 학습 시 사용되는 3가지 메모리 영역
┌────────────────────────────────────────────────────────┐
│ 1. GPU VRAM (비디오 메모리) ── 모델 연산 및 가중치 업데이트  │
├────────────────────────────────────────────────────────┤
│ 2. System RAM (메인 메모리) ── 모델 로드, 가중치 병합 & 변환  │
├────────────────────────────────────────────────────────┤
│ 3. VM Disk (클라우드 디스크) ── 체크포인트 저장 및 라이브러리 설치│
└────────────────────────────────────────────────────────┘
```

### 1. GPU VRAM (비디오 메모리) 초과 (CUDA Out of Memory)
LLM을 학습시킬 때는 모델의 매개변수(Parameter)뿐만 아니라, **역전파(Backpropagation) 계산을 위한 그라디언트(Gradients)**와 **옵티마이저 상태(Optimizer States)**가 모두 GPU 비디오 메모리(VRAM)에 올라가야 합니다.
* **원인**: 0.6B 모델은 매우 가볍지만, 만약 Colab 런타임을 **'T4 GPU'**로 변경하지 않고 기본 **CPU**로 돌렸다면 연산 속도가 극단적으로 느려져 7시간 동안 끝나지 않았을 것입니다. 
* 또한 GPU 모드로 켰더라도, 배치 사이즈(Batch Size)를 과도하게 크게 잡았거나 그래픽 카드의 한계를 넘어서는 텐서 연산이 발생하면 `RuntimeError: CUDA out of memory`가 발생합니다.

### 2. System RAM (시스템 메모리) 초과로 인한 세션 강제 종료
Google Colab 기본 요금제(무료 버전)는 **약 12.7 GB의 System RAM**을 제공합니다.
* **원인**: 파인튜닝 자체는 GPU VRAM에서 무사히 끝났더라도, 학습이 완료된 LoRA 가중치(Adapter)를 원래의 베이스 모델과 병합(Merge)하고 이를 GGUF 형식으로 변환하는 과정(`llama.cpp` 스크립트 실행 등)에서 베이스 모델 전체를 CPU 메인 메모리에 로드하게 됩니다. 
* 이때 시스템 메모리가 12.7 GB를 넘어가면서 Colab 세션이 터지고 **"런타임이 작동 중지되었습니다(RAM을 모두 사용함)"** 메시지와 함께 학습 데이터가 유실되었을 가능성이 큽니다.

### 3. Google Colab 자체 세션 끊김 (Inactivity Timeout)
밤 10시부터 새벽 5시까지 브라우저 창을 닫아두었거나 PC가 절전 모드로 들어갔다면 Colab의 **자동 연결 차단(Timeout)**에 걸려 실패했을 수 있습니다.
* Colab 무료 버전은 사용자의 실시간 인터랙션이 없으면 최대 30분~1시간 이내에 세션을 강제 종료합니다.

> [!NOTE]
> **내 컴퓨터 C드라이브 용량(44.1 GB)은 충분한가요?**
> 네, 충분하고도 넘칩니다! 최종 타겟인 `roseball369/luda-brain-v2` 모델은 `Qwen3-0.6B` 기반에 4비트 양자화(`q4_k_m`)를 거치면 최종 파일 크기가 **약 350MB~450MB 내외**에 불과합니다. 로컬 드라이브에 저장하고 Connect AI 내장 엔진에 바로 로드해 쓰기에 아무런 무리가 없는 가벼운 용량입니다.

## 🛠️ 오늘(7/8) 재도전을 위한 4단계 행동 수칙

어제의 실패를 발판 삼아, 오늘 저녁 재시도 시에는 다음의 수칙을 적용해 '제2의 두뇌 합치기' 프로젝트를 완수하겠습니다.

### Step 1. 런타임 유형 'T4 GPU' 설정 확인하기
Colab 메뉴에서 `런타임` > `런타임 유형 변경`으로 들어가 **T4 GPU**가 반드시 선택되어 있는지 확인합니다. CPU 모드로 돌리면 7시간이 아니라 20시간을 돌려도 끝나지 않습니다.

### Step 2. 메모리 절약을 위한 파라미터 튜닝
* **BitsAndBytes 4비트 로드**: 베이스 모델을 불러올 때 `load_in_4bit=True` 옵션을 주어 GPU VRAM 사용량을 1/4로 줄입니다.
* **Gradient Accumulation**: GPU 메모리가 넉넉지 않다면 Batch Size는 `1` 또는 `2`로 낮추는 대신, `gradient_accumulation_steps`를 `4` 정도로 주어 학습 안정성을 꾀합니다.

### Step 3. 병합(Merge) 및 GGUF 변환 시 메모리 해제
LoRA 어댑터를 기본 모델과 합칠 때, 이전에 학습을 위해 사용했던 불필요한 캐시와 모델 인스턴스를 메모리에서 완벽히 날려주어야 합니다.
```python
import torch
import gc

# 학습용 모델 변수 메모리 해제
del model
del trainer
gc.collect()
torch.cuda.empty_cache()
```
이후 CPU RAM을 아끼기 위해 가중치를 새롭게 불러와 가벼운 상태에서 `merge_and_unload()`를 수행하고 GGUF 변환을 진행합니다.

### Step 4. Colab 연결 유지하기 (절전 모드 방지)
학습이 진행되는 동안 컴퓨터가 잠자기 모드나 절전 모드로 들어가지 않도록 설정하고, 크롬 브라우저 탭이 활성화된 상태를 유지합니다.
